js与PHP数据交互转码问题

工作总结
  • PHP 交互 转码

问题：

  1. ajax向PHP传递XML格式数据中文乱码
  2. XML标签内容为html格式等带特殊符号的复杂内容影响后台PHP校验获取数据
  3. PHP收集XML标签里面内容并在头部和尾部加入预先协定中文生成的md5校验码与前端生成的md5无法确保相同

思路：

  1. 前端对XML标签里面的内容进行编码,后台PHP解析XML对应标签内容进行解码存入数据库
  2. 由于XML标签里面的内容已经是经过一次编码，前端获取到编码的内容在头尾加入协定中文生成md5时又涉及到中文乱码，所以进行二次编码
  3. 后台PHP解析XML对应标签内容在头尾加入协定中文同样先进行二次编码再生成md5与前端的进行校验对比

相关：

  1. escape不编码字符有69个：*，+，-，.，/，@，_，0-9，a-z，A-Z （ECMAScript v3 反对使用该方法）
  2. encodeURIComponent不编码字符有71个：!， ‘，(，)，*，-，.，_，~，0-9，a-z，A-Z
  3. encodeURI不编码字符有82个：!，#，$，&，’，(，)，*，+，,，-，.，/，:，;，=，?，@，_，~，0-9，a-z，A-Z
  4. PHP的urlencode和rawurlencode编码规则：除了“-_.”之外的所有非字母数字字符都将被替换成百分号“%”后跟两位十六进制数
  5. PHP的urlencode和rawurlencode编码区别：urlencode将空格编码为加号“+”，rawurlencode将空格编码为加号“%20”

实际操作：

  1. 前端先用encodeURI对XML标签里面的内容进行第一次编码，头尾加入前后端协定的中文再使用encodeURI进行二次编码，两次编码对一些特殊字符并未编码，虽然PHP的urldecode和rawurldecode方法在解码入库时没有问题，但在编码生成md5却会出现问题。原因是PHP的urlencode和rawurlencode都会对特殊字符进行编码所致
  2. 前端先用encodeURI对XML标签里面的内容进行第一次编码，二次编码时使用encodeURIComponent进行编码再生成md5，第二次编码对一些特殊字符进行了编码，也包括第一次转码后的’%’编码成了’20%’，PHP的urldecode和rawurldecode方法在解码入库时没有问题，在对前端encodeURI方法第一次编码的内容使用urlencode进行再编码时将’%’编码成了’+’导致生成的md5不一样
  3. 基于2，PHP对前端encodeURI方法第一次编码的内容使用rawurlencode进行再编码时生成的md5就一样，原因是rawurlencode方法和js的encodeURIComponent方法一样把’%’编码成了’20%’
  4. 前端两次编码都使用encodeURIComponent编码，PHP解码使用两种解码都行，编码时使用rawurlencode.

依旧bug：

虽然上面的方法转码了大部分特殊符号，但是无论测试还是上线之后都难免出现特殊符号导致各种问题，随着问题的出现最终还是换了另一种方法base64转码。

base64转码

  • 优点
    1. 可以对所有特殊字符进行转码解决了特殊符号带来的困恼
    2. 同时也解决了中文乱码的问题
  • 缺点
    1. js没有现成的方法，不过可以在网上找

总结

如果涉及到文本框输入等复杂的内容数据传递为了避免中文乱码以及各种特殊符号带来的困扰就使用base64转码或者Unicode转码传递。
如果只是URL里面的传递简单的参数可以使用encodeURI和encodeURIComponent等转码。
